# Authentication Flow Documentation

เอกสารนี้อธิบายถึงลำดับการทำงาน (Flow) ของระบบ Authentication ที่ใช้งานในแอปพลิเคชันทั้งหมด ซึ่งขับเคลื่อนโดยระบบ **JSON Web Tokens (JWT) แบบไม่มีสถานะ (Stateless)** และจัดส่งโทเค็นผ่าน **HttpOnly cookies** เพื่อความปลอดภัยสูงสุด อีกทั้งยังเสริมด้วยกลไกการจำกัดสิทธิ์ **Role-Based Access Control (RBAC)** พร้อมด้วยระบบเปิดรับรหัส 2 ขั้นตอน **(TOTP 2FA)**

---

## 🏗️ Core Mechanisms
- **`jwt` (Access Token):** โทเค็นอายุสั้นสำหรับใช้เข้าถึงระบบต่างๆ ภายใน (มีอายุ 15 นาที) จัดเก็บอยู่ในระบบแบบ `httpOnly`, การรันบน production จะใช้ตัวเลือก `secure`, ตั้งค่า `sameSite='lax'` และกำหนดเส้นทางให้วิ่งผ่านแบบ `path='/'` เสมอ
- **`refreshToken`:** โทเค็นอายุยาว (7 วัน) สำหรับขอต่ออายุการทำงานของ `jwt` ใช้งานเฉพาะบนพาธ `path='/auth'` ท้ายที่สุดเพื่อป้องกันการถูกเผยแพร่หรือรั่วไหลไปยัง Endpoint อื่นที่ไม่ได้เกี่ยวข้องเลย
- **Cryptography:** มีการใช้ศาสตร์แห่งการเข้ารหัส `argon2` สำหรับการแฮชรหัสผ่าน และใช้ไลบรารี `otplib` เป็นตัวเอกในการตั้งค่าชุดรหัส TOTP สองขั้นตอนแบบจำกัดเวลา

---

## 🔄 Authentication Flows

### 1. Traditional Email/Password Login
1. **Client Action:** ส่งข้อมูลได้แก่ `email` และ `password` วิ่งตรงไปยังเป้าหมาย `POST /auth/login`
2. **Server Action:** ทำการตรวจสอบและเทียบรหัสผ่านจากฐานข้อมูลด้วยการรันคำนวณ Hashing
3. **2FA Check:**
   - **กรณีที่ 2FA ถูกปิดไว้:** เซิร์ฟเวอร์จะคืนโทเค็น `jwt` และ `refreshToken` ใส่เข้ามาในคุกกี้ที่โดนล็อคไว้แบบ HttpOnly พ่วงไปถึงการคืนรหัสสถานะ `200 OK` ถือเป็นการสิ้นสุดกระบวนการ
   - **กรณีเปิดใช้งาน 2FA:** ระบบจะระงับการสร้างคุกกี้ Session ตัวจริง แล้วพ่นตอบกลับแนบไปแทนว่า `{ requiresTwoFactor: true, tempToken: "xxxx" }` ชั่วคราว

### 2. Two-Factor Authentication (2FA) Login
1. **Client Action (Pending 2FA):** ผู้ใช้ได้รับตัวแปร `tempToken` มาจากสเตปแรก และต้องพิมพ์ตัวเลขแบบ TOTP 6 อักขระจากแอปพลิเคชันยืนยันตัวตน
2. **Submission:** ส่งคำร้องพ่วงไปพร้อม `code` คละเคล้ากับ `tempToken` ไปประเมินผลที่ `POST /auth/2fa/verify-login`
3. **Server Action:** หากตรวจรับแล้วตัวแปร `tempToken` ปลอดภัยและโค้ดตัวเลขนี้เชื่อถือได้ ระบบถึงจะฉีดสร้างคุกกี้ Session ของจริง (`jwt` / `refreshToken` แบบ HttpOnly) ลงให้บราวเซอร์

### 3. Google OAuth 2.0 Registration/Login
1. **Initiation:** ผู้ใช้เลือกยิงพิกัดไปที่ `GET /auth/google` หลังนั้นเซิร์ฟเวอร์จะรีไดเรกต์ไปดึงหน้าจอความยินยอมตามฉบับของบัญชี Google
2. **Callback:** Google ตีกลับข้อมูลส่งรับที่สอดคล้องกันมาที่ `GET /auth/google/callback` พร้อมประวัติส่วนตัวผู้ใช้ระดับพื้นฐาน
3. **Processing:**
   - Auth Guard ทำหน้าที่ดึงข้อมูลอย่างละเอียดเฉพาะ `email` และ `name` เข้ามาประมวลผล
   - หากผู้ใช้ยังไม่เคยมีบันทึกในระบบ ระบบจะก่อร่างสร้างบัญชีอัตโนมัติ แต่หากมีแล้วจะเปิดวาร์ปพาเข้าคลังบัญชีนั้นต่อ
   - ระบบตั้งค่าสล็อตคุกกี้ แจก `jwt` พร้อม `refreshToken` ประทับรอยไปให้ชั่วพริบตาเดียว
4. **Redirection:** ผลิกลำนำสู่การเปลี่ยนทิศทาง (302 Redirect) โยนผู้ใช้เข้าไปถึงโซนพื้นที่ทำงาน `/dashboard/personal` บนฝั่งหน้าตาโปรแกรมที่สร้างจาก Vite ทันที

### 4. Registration Process
1. **Submission:** ข้อมูลส่วนสำคัญประกอบไปด้วยบัญชี (ชื่อ, อีเมล, รหัสผ่าน, ตำแหน่งแผนก) ย้ายไหลไปบรรจบสู่ `POST /auth/register`
2. **Creation:** โครงแผงระบบประดิษฐ์ขึ้นเป็นผู้ใช้รายใหม่ตามปกติและกำหนด บทบาทสิทธิ์เริ่มต้นโดยออโต้
3. **Persistence:** ดำเนินเปิดระบบแผงบัญชีทันที โดยระบบแนะคุกกี้พ่วง `jwt` / `refreshToken` ให้กับผู้ใช้งาน ช่วยให้พวกเขาเข้าสู่ความลื่นไหลไม่ต้องคลิกปุ่มล็อกอินซ้ำอีก

## 🛡️ Session Lifecycle Management

### Token Refreshing (Silent Session Renewal)
1. เมื่อ `jwt` ถูกผลาญเวลาหมดลงแล้ว (หลังผ่านไปประมาณ 15 นาที) ทุกการตอบสนองร้องขอระหว่างฝั่งเว็บถึง Backend ย่อมตกสู่ปัญหารหัสแจ้งเตือน `401 Unauthorized` ในแวบแรก
2. มิดเดิลแวร์ผู้ดูแลในบราวเซอร์ (Interceptor) จะขโมยซีนกระบวนท่าไปสั่งการ `POST /auth/refresh` ลับหลังผู้ใช้เองอย่างเงียบๆ
3. เซิร์ฟเวอร์จะหยิบตรวจคุกกี้เฉพาะที่มีคลังของ `refreshToken` ออกมาคลี่ประเมินความปลอดภัยทันที
4. **Rotation:** หากทุกอย่างลงล็อกไร้มลทิน เซิร์ฟเวอร์จะทบยอดชุดกระตุกกลับเป็น `jwt` ฉบับปรับปรุงใหม่และอัปเดตเปลี่ยนผ่าน `refreshToken` ทดแทนเป็นชุดใหม่ (หมุนเวียนสลับเปลี่ยน) มอบความสดใหม่คืนให้บราวเซอร์ทำหน้าที่ได้ยาวนาน

### Logout Process
1. เรียกใช้การยกเลิกกระบวนท่า `GET /auth/logout` เมื่อต้องการเก็บข้าวของ
2. ฝ่ายดูแลระบบจะสั่งยับยั้งสิทธิ์คุกกี้ของ `refreshToken` โดยลบพรมแดนทั้งหมดออกจากสารบบผู้เช่าของ Database เพื่อปัดตกการประเมินยืนยันผลต่อไป
3. ฝ่ายระบบยืนยันใช้งานฟังก์ชัน `res.clearCookie()` เพื่อประกาศลบล้างชะล้างความทรงจำคุกกี้ที่เก็บไว้ `jwt` และ `refreshToken` ไปเกลี้ยงหมดในทันที

## ⚙️ 2FA Management (Requires Active Session)
- **Generate:** คิวรี่ `POST /auth/2fa/generate` คือประตูระบุสร้างแกนเคล็ดลับ (Secret key) ที่คืนค่ารูปภาพเพื่อการนำไปแสกน QR Code ของแอปตัวช่วย (อาทิ Google Authenticator)
- **Enable:** ระบบรังหนุน `POST /auth/2fa/enable` ขอยกระดับเงื่อนไขจากเลขโค้ดที่ระบบตัวช่วยให้มา พิมพ์ให้ครบตามที่ระบุเพื่อยืนยันการตั้งค่าสำเร็จ 
- **Disable:** ระบบปอกสิทธิ์ `POST /auth/2fa/disable` เป็นด่านชัตดาวน์คำสั่งปิดพ่วงด้วยเงื่อนไขพิสูจน์ยืนยันผ่านรหัสปัจจุบันอีกชั้นเพื่อความปลอดภัยสูงสุด

---

## 🔒 Password Recovery Layer
- **Forgot Password:** คิวรี่สู่ช่องทาง `POST /auth/forgot-password` ดงระบำปั่นรหัสสุ่มผสมเป็นแผ่นโทเค็นเฉพาะกิจถูกแจกจ่ายพ่นตรงเข้าสู่ทางจดหมายอีเมลหลักของฝ่ายผู้ประสงค์ใช้
- **Reset Password:** กิจกรรมรัน `POST /auth/reset-password` ต้องประชันควบคู่ไปกับชุดรหัสนั้น และจังหวะรีเซ็ตใหม่ ตัวระบบมักจะสั่งเบรกเครือข่ายความยึดโยง Session โดยใช้อำนาจลักล้างการล็อกอินอันตรายใดๆ ออกทั่วพิกัดคุกกี้เดิมทิ้งอย่างเบ็ดเสร็จ **(Global Cookie Clear)**
