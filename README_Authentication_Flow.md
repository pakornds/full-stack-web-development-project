# กระแสการพิสูจน์ตัวตนและการทำงานของฝั่ง Authentication (Authentication Flow)

เอกสารฉบับนี้อธิบายลำดับชั้นและกระบวนการเข้าใช้งานของระบบ Leave Management System โดยจะมุ่งเน้นไปที่ขั้นตอนการพิสูจน์และยืนยันตัวตน (Authentication) เป็นหลัก

---

## 1. กระบวนการลงทะเบียน (Registration Flow)
1. **ผู้ใช้กรอกข้อมูล:** อีเมล, ชื่อ-นามสกุล, และรหัสผ่านที่หน้าลงทะเบียน
2. **การตรวจสอบช่องโหว่:** ฝั่ง Backend (`RegisterDto`) มีการใช้ `class-validator` ตรวจสอบรูปแบบอีเมลและความยาวรหัสผ่าน
3. **การเข้ารหัส:** รหัสผ่านจะถูกเข้ารหัสระดับสูงด้วย **argon2** ก่อนบันทึกลงในฐานข้อมูล 
4. **การเตรียมระบบเบื้องต้น:** ระบบลงบันทึกจำนวนโควต้าการลามาตรฐานในตาราง `LeaveQuota` ให้โดยอัตโนมัติ
5. **การสร้าง Token ขั้นต้น:** เมื่อบันทึกเสร็จสิ้น ระบบจะจ่าย JWT และ Refresh Token เพื่อให้ผู้ใช้เข้าสู่ระบบได้ทันที

---

## 2. กระบวนการล็อกอินปกติ (Standard Login Flow)
1. **รับข้อมูล (Credentials):** ผู้ใช้งานป้อนอีเมลและรหัสผ่าน
2. **Anti-Bruteforce & Account Lockout:** Backend จะเช็คค่า `lockedUntil` ก่อน 
   - หากบัญชีถูกล็อกเนื่องจากกรอกรหัสผิดครบ 5 ครั้ง (`failedLoginAttempts >= 5`) ระบบจะจำกัดไม่ให้ล็อกอินเป็นเวลา **15 นาที**
3. **การเปรียบเทียบรหัสผ่าน:** Backend ตรวจสอบรหัสผ่านกับค่า Hash (Argon2) แบบ Native
4. **ตรวจสอบสถานะ 2FA:** 
   - **กรณีเปิดใช้งาน 2FA:** ระบบจะยัง **ไม่แจก Access Token** แต่จะส่ง `tempToken` (Token ชั่วคราว) กลับไป เพื่อพาผู้ใช้เข้าไปสู่หน้ากรอกรหัส TOTP ขั้นถัดไป
   - **กรณีไม่ได้เปิด 2FA:** ระบบจะทำการสุ่ม `sessionId` และผูกเก็บเข้าตารางฐานข้อมูลผู้ใช้งาน (`currentSessionId`) จากนั้นออก **Access Token** และ **Refresh Token** และจบการล็อกอินได้สำเร็จ

---

## 3. กระบวนการยืนยันตัวตนแบบหลายปัจจัย (2FA / TOTP Flow)
**ช่วงตั้งค่า (Setup):**
1. ผู้ใช้งานเรียก API ตั้งค่า 2FA ระบบจะสร้าง *Secret* ผ่านไลบรารี `otplib` พร้อมส่ง QR Code ให้สแกนบนเครื่องมือ (เช่น Google Authenticator)
2. ผู้ใช้สแกนและนำรหัส 6 หลักมากรอกยืนยันอีกครั้ง หากผ่าน ค่า `twoFactorEnabled` จะเปลี่ยนเป็น `true` และบันทึก Secret ในตาราง

**ช่วงล็อกอินผ่าน 2FA (Challenge):**
1. เมื่อผู้ใช้ล็อกอินรหัสผ่านถูก (จากขั้นตอนที่ 2) นำค่า `tempToken` พร้อมตัวเลข 6 หลักมากรอกที่ระบบ
2. เซิร์ฟเวอร์ตรวจสอบเลข 6 หลักกับ `twoFactorSecret` ที่เก็บไว้ หากถูกต้อง ระบบจึงจะออก Access Token และ Refresh Token ตัวจริงให้ใช้งาน

---

## 4. กระบวนการ Refresh Token (Token Rotation)
1. ข้อมูล `Refresh Token` ภายในฐานข้อมูลไม่ได้เก็บเป็นสายอักขระเพียว (Plaintext) แต่จะถูกเก็บรวบยอดเช็คด้วยการแฮชแบบ `hashRefreshToken()` 
2. เมื่อ Access Token ใกล้หมดอายุ ฝั่ง Frontend จะยิงการขอ Token ใหม่
3. **การตรวจสอบ:** ระบบประเมิน 3 จุด: 
   - Session ID ของ Token กับฐานข้อมูลยังตรงกันหรือไม่
   - Refresh Token Hash ถูกต้องหรือไม่
   - วันที่ยังไม่หมดอายุใช่หรือไม่ 
4. หากผ่านเงื่อนไข ระบบจึงจะแจก Token ชุดใหม่ให้ใช้งานต่อ

---

## 5. กระบวนการผ่านผู้ให้บริการยืนยันตัวตนภายนอก (OAuth 2.0 Flow - Google)
1. ผู้ใช้กดปุ่ม **“Login with Google”**
2. ระบบพ่นไปที่หน้าต่างยินยอม (Consent screen) จาก Google เมื่ออนุญาต ระบบ Google จะตีกลับนำข้อมูล (Email และ Name) เข้าสู่ระบบ Backend
3. Backend ตรวจสอบ Email (หากไม่มีจะสร้าง User ใหม่และเจาะ Default Quota ให้) จากนั้นออก Access Token และบันทึก Session เข้าฐานข้อมูลตามปกติโดยข้ามการขอรหัสผ่านไป

---

## 6. กระบวนการรีเซ็ตรหัสผ่าน (Password Reset Flow)
1. ผู้ใช้งานกดเมนูลืมรหัสผ่าน (Forgot Password) พร้อมกรอกอีเมล
2. **Anti-Enumeration:** ไม่ว่าอีเมลจะมีอยู่ในระบบหรือไม่ ระบบจะตอบกลับแบบกว้างว่า "หากมีอีเมลนี้ ลิงก์รีเซ็ตถูกส่งไปแล้ว" เพื่อป้องกันการแฮกข้อมูล (Data scraping)
3. หากมีตัวตนจริง, Token รีเซ็ตแบบ UUID กึ่งเข้ารหัสจะถูกสร้างขึ้นเพื่อหมดอายุใน 15 นาที พร้อมส่งเข้า Email ด้วย **Nodemailer**
4. ผู้ใช้คลิกลิงก์ กดยืนยันด้วย Token ปัจจุบัน ระบบต้อนรับรหัสผ่านใหม่และทำการเคลียร์ `currentSessionId` บนฐานข้อมูล พร้อมกับเซ็ตให้ `failedLoginAttempts` กลับเป็นศูนย์ ถือเป็นการเตะระบบล็อกอินและบังคับให้ผู้ใช้ใส่รหัสผ่านใหม่เข้าสู่ระบบอีกครั้ง

---

## 7. กระบวนการออกจากระบบ (Logout Flow)
1. ผู้ใช้กดคำสั่งออกจากระบบ
2. ค่า `currentSessionId` ในฐานข้อมูล รวมถึง Token Hash จะถูกลบ (`null`) ให้หมด
3. เซสชันต่างๆ บนเบราว์เซอร์จะใช้งานต่อไปไม่ได้อีก จึงสรุปขั้นตอนจบการทำงานในรอบการใช้งานได้อย่างสมบูรณ์
