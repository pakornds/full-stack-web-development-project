# Security Measures (OWASP Top 10 Mapping)

โปรเจกต์นี้ได้รับการพัฒนาโดยเน้นแนวคิด "Security by Design" (ความปลอดภัยตั้งแต่การออกแบบ) เพื่อให้มั่นใจว่าช่องโหว่ที่สำคัญจะถูกป้องกันและจัดการตั้งแต่ระดับโครงสร้างหลักของแอปพลิเคชัน เอกสารนี้เป็นการเชื่อมโยงมาตรการรักษาความปลอดภัยของระบบเข้ากับความเสี่ยงเด่นๆ ของเว็บแอปพลิเคชันตามมาตรฐาน **OWASP Top 10 (2021)**

---

## 🛡️ OWASP Top 10 Mapping

### A01:2021 - Broken Access Control
ป้องกันมิให้ผู้ใช้สามารถทำสิ่งที่อยู่นอกเหนือสิทธิ์ที่ได้รับ
- **Role-Based Access Control (RBAC):** มีการใช้ `@Roles()` decorator ใน NestJS ควบคู่กับ custom `RolesGuard` อย่างเคร่งครัด เพื่อตรวจสอบสิทธิ์ในแต่ละ Endpoint (เช่น พนักงานปกติไม่สามารถดูคำขอลาของแผนกอื่นได้นอกจากจะมีสิทธิ์เป็นผู้จัดการหรือแอดมิน)
- **Service-Level Verification:** ภายใน `LeavesService` จะมีการป้องกันในระดับตรรกะของอ็อบเจ็กต์อีกชั้น เช่น พนักงานจะได้รับอนุญาตให้ใช้คำสั่ง `update` หรือ `remove` ได้เฉพาะคำขอร้องของตนเองเท่านั้น และจะถูกปฏิเสธทันทีหากสถานะไม่ได้เป็น `pending`
- **CORS Configuration:** ถูกตั้งค่าอย่างรัดกุมใน `main.ts` โดยระบุ `origin` เฉพาะเจาะจงที่ชี้ไปยัง Frontend URL ควบคู่กับการอนุญาตแบบ `credentials: true`

### A02:2021 - Cryptographic Failures
ปกป้องข้อมูลที่ละเอียดอ่อนทั้งระหว่างเก็บและระหว่างการย้ายโอน
- **Password Hashing:** ไม่มีการเก็บรหัสผ่านในรูปแบบตัวอักษรธรรมดาอย่างเด็ดขาด แต่ใช้ **Argon2** ซึ่งเป็นเทคนิคการเข้ารหัสและกระจายค่าแฮชที่แน่นหนาและได้รับการยอมรับว่าปลอดภัยกว่า bcrypt
- **2FA / MFA Support:** ระบบรองรับการตั้งค่าการยืนยันตัวตนสองชั้นผ่าน `otplib` และรูปแบบ Time-based One-Time Passwords (TOTP) ช่วยลดความเสี่ยงจากการขโมยรหัสผ่าน (Credential Stuffing)
- **JWT Standard:** ใช้ JSON Web Tokens (JWT) ในการระบุตัวตน ซึ่ง Token ทุกชุดได้รับการเข้ารหัส/เซ็นกำกับ (Sign) ไว้อย่างปลอดภัยด้วย `@nestjs/jwt` ทำให้ไม่สามารถแอบแก้ไขข้อมูลใน Token ระหว่างทางได้

### A03:2021 - Injection (Including XSS)
ป้องกันการทำงานของคำสั่ง/สคริปต์ที่ประสงค์ร้าย
- **ORM-based Data Mapping:** จำกัดการคิวรีฐานข้อมูลโดยตรง โดยใช้ **Prisma** ซึ่งทำหน้าที่กลั่นกรองและห่อหุ้มคำสั่งให้มีความปลอดภัยสูง ช่วยป้องกัน SQL Injection อย่างเป็นรูปธรรม
- **Cross-Site Scripting (XSS) Sanitization:** ทุก ๆ ข้อความบรรยายต่างๆ ที่ส่งมาจากฝั่งผู้ใช้ (เช่น เหตุผลในการลา) จะต้องผ่านเครื่องมือ `sanitize-html` ที่ถูกตั้งค่าไว้ที่ระดับความปลอดภัยสูงสุด (`{ allowedTags: [], allowedAttributes: {} }`) ช่วยล้างสคริปต์แอบแฝงทั้งหมดก่อนจะนำไปเซฟที่ฐานข้อมูล

### A04:2021 - Insecure Design
ใช้ระบบที่ป้องกันภัยคุกคามต่างๆ เบื้องต้นตั้งแต่ขั้นตอนเริ่มระบบ
- **Validation Pipeline Setup:** โปรเจกต์ใช้ `class-validator` ในการทำ Global `ValidationPipe` และจุดที่สำคัญคือการเปิดใช้งาน `{ whitelist: true }` ทำให้ Payload หรือฟิลด์ที่ส่งเข้ามาที่ไม่ได้ลงทะเบียน (หรือแฮกเกอร์จงใจส่งมาเกิน) ถูกลบทิ้งทันที
- **Strict Decoupling:** ใช้งานการแบ่งฝั่งระหว่าง NestJS Backend กับ React/Vite Frontend (Decoupled architecture) ทำให้การทำงานของหน้าเว็บใช้รูปแบบ API ทั้งหมด ลดความเสี่ยงในการถูกรันสคริปต์บนแม่แบบของเซิร์ฟเวอร์ (Template Injection) แบบระบบยุคเก่า

### A05:2021 - Security Misconfiguration
เลี่ยงการใช้ค่าเริ่มต้น ตั้งค่าให้แข็งแกร่ง และแอบซ่อนความลับอย่างปลอดภัย
- **Environment Management:** พวกข้อมูลลับ (Secrets) ถูกแยกจัดการทั้งหมดผ่าน `@nestjs/config` และตั้งรันแยกในไฟล์ `.env` ที่ไม่อนุญาตให้เชื่อมต่อตรงๆ จากภายนอก
- **NestJS Defaults:** คุณลักษณะเบื้องต้นของ NestJS ออกแบบมาให้แยกการทำงานออกจากกัน ทำให้ยากต่อการตั้งค่าผิดพลาด

### A06:2021 - Vulnerable and Outdated Components
ใช้เทคโนโลยีใหม่ที่ได้รับการอัปเดตอย่างสม่ำเสมอ
- **Modern Lifecycle:** โปรเจกต์ถูกพัฒนาขึ้นบนฐานของ Nest 11 และ React 19 ช่วยลดปัญหาช่องโหว่ในระดับโค้ดย่อยเดิมลงไปมหาศาล และจัดการความสัมพันธ์รุ่นอย่างเป็นระบบโดย `package-lock.json`

### A07:2021 - Identification and Authentication Failures
การยืนยันบุคคลที่เข้ามาใช้งานให้มีความถูกต้องและไม่รั่วไหล
- **OAuth 2.0 Integration:** มีการผสาน `passport-google-oauth20` สำหรับผูกผู้ใช้เข้ากับกลไกยืนยันตัวที่ได้มาตรฐาน เพื่อการจัดการผู้ใช้ที่สะดวกและปลอดภัย
- **Guarded Tokens:** JWT จะไม่อยู่ในพื้นที่ LocalStorage ของบราวเซอร์ แต่อยู่ในรูปแบบ **HttpOnly Cookies** (ซึ่งถูกสร้างร่วมกับ `cookie-parser`) ไม่สามารถถูกเขียน/อ่านจากโค้ด JavaScript ใด ๆ บนเว็บได้ ป้องกันการถูกขโมยโดยตรงและเสริมเกราะป้องกันต่อสคริปต์ที่ประสงค์ร้าย

### A08:2021 - Software and Data Integrity Failures
หลักประกันรักษาระบบและข้อมูลไม่ให้ซ้อนทับหรือผิดเพี้ยน
- **Entity State Logic Constraints:** ในจุดที่เป็นตรรกะสำคัญอย่างการอนุมัติ (`LeavesService.updateStatus`) และการจัดการแผนงานลา จะใช้ Prisma ควบคุมในระดับ `$transaction` ทุกครั้ง เพื่อรับประกันว่าการบันทึกข้อมูลหลักและการปรับลดโควตาวันหยุด จะสำเร็จพร้อมกันหรือถูกยกเลิกไปพร้อมๆ กัน ไม่ทำให้เกิดภาวะข้อมูลชำรุด (Data Corruption)

### A09:2021 - Security Logging and Monitoring Failures
ระบบติดตามร่องรอยตรวจสอบพฤติกรรมในฐานข้อมูล
- **Comprehensive Audit Log:** มีการใช้งาน `AuditService` แบบภายใน เพื่อรับหน้าที่ติดตามการเปลี่ยนแปลงข้อมูลหลัก (`CREATE`, `UPDATE`, `UPDATE_STATUS`, `DELETE`) ในระบบ ข้อมูลทั้งหมด เช่น อีเมลของผู้กระทำ, ID เอกสาร, และสถานะที่เปลี่ยนไป จะถูกบันทึกไว้อย่างรัดกุมพร้อมตราประทับวันเวลา (Timestamps) เพื่อรองรับการสืบสวนตามหลักการความรับผิดชอบ

### A10:2021 - Server-Side Request Forgery (SSRF)
ป้องกันเซิร์ฟเวอร์โดนสั่งดึงข้อมูลจากแหล่งปลายทางที่ไม่พึงประสงค์
- **External URL Scoping:** การใช้วงจรทิศทางการดึงข้อมูลภายนอกถูกสงวนจำกัดสิทธิ์เฉพาะส่วนของระบบอย่างระมัดระวัง (เช่น Google ผ่าน Passport ไหลเวียนตามโฟลว์ที่กำหนดไว้) ข้อมูล Payload ใดๆ ก็ตามที่ส่งมาเป็น URL จากฝั่งผู้ใช้งาน จะได้รับการประมวลผลให้มองว่าเป็นตัวอักษรธรรมดาและไม่ท้อนคำสั่ง Query หรือ Fetch ไปที่เป้าหมายนั้นเด็ดขาด
